One port, one listener

Lesson 0001 · Many Branches, One Port · ~10 minutes

Mission check: you want a stable branch and feature branches running at once, but OAuth login only works at exactly http://localhost:3000/…. This lesson explains the two mechanical laws that create that trap — so the fix, when it comes, is obvious rather than magic.

Law 1: an (address, port) pair has exactly one owner

"Port 3000" is not a thing a program owns. What a program owns is a listening socket bound to an (address, port) pair — here, (127.0.0.1, 3000). The claim happens at the bind call, before any traffic flows, and the operating system enforces uniqueness: the second process to try gets EADDRINUSE and dies. There is a kernel escape hatch called SO_REUSEPORT that lets several processes share a port — but the kernel then load-balances connections across them arbitrarily. You'd get branch A or branch B at random on each request. Precisely useless for our mission.

This is why the constraint is real and not just a framework quirk: no amount of Next.js or Vite configuration lets two dev servers both be localhost:3000.

Try it — 60 seconds

Watch the OS enforce Law 1. In one terminal:

node -e "require('http').createServer((q,s)=>s.end('branch A\n')).listen(3000,()=>console.log('A owns (127.0.0.1, 3000)'))"

In a second terminal, try to start "branch B" the same way:

node -e "require('http').createServer((q,s)=>s.end('branch B\n')).listen(3000)"
# → Error: listen EADDRINUSE: address already in use :::3000

See who the owner is (this command is worth memorizing):

lsof -nP -iTCP:3000 -sTCP:LISTEN

Law 2: redirect_uri matching is exact — by design, everywhere

When your app sends the browser to the identity provider, it includes a redirect_uri — where the browser should be sent back with the authorization code. The provider compares it against the registered value using simple string comparison: character-for-character equality. This isn't your provider being fussy in dev mode; it's RFC 9700 §2.1, the OAuth security Best Current Practice: servers MUST use exact matching, because every pattern-matching scheme ever deployed has been exploited to steal authorization codes (§4.1). The one exception: native apps may vary the port on loopback URIs (RFC 8252 §7.3), because a desktop binary can't guarantee any port is free. Browser apps don't qualify — which is exactly the wall you hit.

Run the registered value http://localhost:3000/callback against the obvious workarounds and each fails on a different character:

http://localhost:3000/callback   registered ✓
http://localhost:3001/callback   port differs        ✗
http://127.0.0.1:3000/callback   host differs        ✗
http://feat.localhost:3000/...   host differs        ✗
https://localhost:3000/callback  scheme differs      ✗

So: Law 1 says only one process can answer at localhost:3000. Law 2 says the login flow must pass through localhost:3000. Trap closed — apparently.

The reframe that opens the trap

Read Law 2 again, precisely. It constrains what the browser and the identity provider see: one exact URL. It says nothing about what happens behind that URL. And Law 1 says one listener — it never says the listener must be your app.

Nothing requires the process at localhost:3000 to be a branch. It can be a doorman — a reverse proxy — that forwards each request to any of N branches running on internal ports the browser never sees.

Branch A runs on 3101, branch B on 3102, the proxy owns 3000. Every URL in the OAuth dance — including the callback — is localhost:3000, so both laws are satisfied, yet N branches are alive at once. The remaining puzzle (next lesson) is how the doorman decides which branch gets a given request — that's where routing, cookies, and browser profiles come in.
