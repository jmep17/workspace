The doorman

Lesson 0002 · Many Branches, One Port · ~15 minutes, hands-on

Mission check: lesson 0001 ended with a claim — the process at localhost:3000 doesn't have to be a branch, it can be a doorman. Today you build that doorman and swap branches behind it while the browser-visible origin never changes. By the end you'll have the "stable branch + instant switch" half of your mission working in miniature.

Warm-up: retrieve before you build

From memory, before reading on — this is what makes lesson 1 stick.

What a reverse proxy actually does

A reverse proxy is an ordinary HTTP server that answers every request by making a second, separate HTTP request of its own. One proxied page load involves two TCP connections: browser → proxy (the front door), and proxy → upstream (a branch on an internal port). The proxy copies the request across, relays the response back, and the browser is none the wiser — its connection began and ended at localhost:3000. "Reverse" distinguishes it from a forward proxy, which sits in front of clients (like a corporate web filter). A reverse proxy sits in front of servers. Same copying trick, opposite side of the conversation.

The copy is not byte-identical: the proxy annotates it. Caddy's reverse_proxy passes the original Host header through and adds X-Forwarded-For (the real client address) and X-Forwarded-Proto (the original scheme), so the branch can still know who really called. You'll see these headers with your own eyes in the lab — the upstream servers below echo every header they receive.

One trap before you start: automatic HTTPS

Caddy's signature feature is automatic HTTPS: give it a site address with a recognizable hostname — even localhost:3000 — and it will serve https with a locally-generated certificate. Normally delightful. For us, fatal: you learned in lesson 1 that the scheme is part of the exact-match string, and https://localhost:3000/callback ≠ http://localhost:3000/callback. The fix is to state the scheme explicitly in the site address: http://localhost:3000. Mechanics, not incantation — you now know exactly why that prefix must be there.

Lab: build the doorman

Step 0 — Install Caddy (once)

brew install caddy
caddy version

Checkpoint: a version string like v2.x.x.

Step 1 — Two fake branches

Terminal 1 — "stable", on internal port 3101:

node -e "const h=require('http');h.createServer((q,s)=>{s.end('STABLE BRANCH (3101)\n\n'+JSON.stringify(q.headers,null,2)+'\n')}).listen(3101,()=>console.log('stable up on 3101'))"

Terminal 2 — "feature", on internal port 3102:

node -e "const h=require('http');h.createServer((q,s)=>{s.end('FEATURE BRANCH (3102)\n\n'+JSON.stringify(q.headers,null,2)+'\n')}).listen(3102,()=>console.log('feature up on 3102'))"

Checkpoint: curl -s localhost:3101 says STABLE, curl -s localhost:3102 says FEATURE. Two branches, alive at once — legal because they're on different (address, port) pairs.

Step 2 — The doorman takes the front door

Terminal 3:

mkdir -p ~/src/workspace/labs/doorman
cd ~/src/workspace/labs/doorman
printf 'http://localhost:3000\n\nreverse_proxy :3101\n' > Caddyfile
caddy run --config Caddyfile

Checkpoint: in a fourth terminal, curl -s localhost:3000 prints STABLE BRANCH (3101) — served from port 3000. Look at the echoed headers: X-Forwarded-For: 127.0.0.1 and X-Forwarded-Proto: http are the proxy's annotations; Host is still localhost:3000. The upstream knows it's behind a doorman; the browser doesn't.

Step 3 — Swap branches without touching the origin

Edit the Caddyfile so the upstream is :3102, then hot-reload:

printf 'http://localhost:3000\n\nreverse_proxy :3102\n' > Caddyfile
caddy reload --config Caddyfile

Checkpoint: curl -s localhost:3000 now prints FEATURE BRANCH (3102). Same port, same scheme, same host — an OAuth provider watching this flow could not tell anything changed. You just swapped the entire application behind a login-safe origin, with zero downtime.

Open http://localhost:3000 in your browser too — worth seeing outside curl.

Docker variant — the whole lab, no brew

Prefer containers? The lab lives pre-built in labs/doorman-docker/ (a compose file, the Caddyfile, and one shared branch server script). The full command sequence is also in that folder's README.md for terminal-side copying:

cd ~/src/workspace/labs/doorman-docker
docker compose up

Checkpoints: curl -s localhost:3000 says STABLE — but curl -s localhost:3101 now fails: the branches publish no ports, so they are reachable only through the doorman. To swap branches, edit the Caddyfile's upstream to feature:3102 and hot-reload the running container:

docker compose exec doorman caddy reload --config /etc/caddy/Caddyfile

(caddy reload must run inside the doorman container — that's where the admin endpoint lives — hence docker compose exec.)

Two mechanics to notice. First, the Caddyfile says stable:3101, not localhost:3101 — inside the container, localhost means the container itself; loopback is per network namespace, and compose gives services DNS names instead. Second, ports: "127.0.0.1:3000:3000" is Docker's own little doorman: a forwarder from the host's (127.0.0.1, 3000) into the container. Your browser still sees exactly http://localhost:3000, so both laws hold. Later, when the upstreams are real npm run dev processes on your Mac rather than containers, a dockerized doorman reaches them as host.docker.internal:3101 — Docker Desktop's name for "the machine hosting me".

The origin is a contract with the browser and the IdP. The upstream is an implementation detail. Caddy lets you re-decide the implementation detail at any moment, without renegotiating the contract.

Notice what you already have: keep stable on 3101 and any feature branch on 3102, and caddy reload is a one-line "which branch is live?" switch — OAuth intact. What you don't have yet is both at once in two browser windows. That needs the doorman to route per session rather than for everyone — and cookies make that subtle. That's lesson 0003.
