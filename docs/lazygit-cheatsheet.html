Cherry-picking

  1. In any commits view (including another branch's, via Branches → enter), press C on each commit to copy it (range-select with v works too).
  2. Checkout the target branch.
  3. In its commits panel press V to paste. esc clears the copied set.

Custom patches — "how to patch"

Pull specific files or lines out of an existing commit (split a commit, move code between commits, extract to index):

  1. Commits panel → enter on the commit to see its files.
  2. Build the patch:
    • space on a file → toggle the whole file into the patch (green = whole file, yellow = partial), or
    • enter on a file → select lines (v for ranges, a for hunks) and space to toggle them into the patch.
  3. Press ctrl+p to open the custom patch options menu:
    • Remove patch from commit — deletes those changes from the commit (they vanish entirely)
    • Move patch out into index — pulls the changes out of the commit and leaves them staged
    • Move patch into new commit — splits the commit, placing the patch in a new commit after it
    • Move patch to selected commit — select another commit first, then move the changes into it
    • Apply patch / apply in reverse — apply to the working tree
    • Copy patch to clipboard

If a step causes conflicts, resolve them (see below) and continue with m. ctrl+p also shows "reset patch" to start over.

Undo only some lines in a file from a previous commit — all the ways

The situation: a few commits ago you committed a file, and some of those lines were a mistake — but the rest of the commit (and the commits after it) are fine. There are five ways to fix this, split into two families: methods that rewrite history (the mistake disappears as if it never happened, but commits get new SHAs, so a pushed branch needs a force-push) and methods that add a new commit on top (the mistake stays visible in history, but nothing is rewritten — always safe on shared branches). Unsure which commit introduced the lines? ctrl+s → filter by path, or launch with lazygit -f <file>.

Method 1 — Custom patch builder (rewrites history; the most surgical)

What it means: you select the exact lines inside the old commit, lazygit builds a patch (a mini-diff) from them, then runs an interactive rebase behind the scenes that re-creates the commit without those lines.

  1. Commits panel (4) → select the commit → enter to see its files → select the file → enter to view its diff.
  2. Select only the lines to undo: move with j/k, v to start a range selection, then space to add them to the patch. (a grabs a whole hunk; space toggles lines back out.)
  3. esc, then ctrl+p → remove patch from commit. Done — the commit now exists without those lines. (Or move patch out into index if you want the removed changes handed back to you as staged changes instead of deleted.)
  4. If later commits touched the same lines you'll get conflicts — resolve, then m → continue.

Method 2 — Edit the commit mid-rebase (rewrites history; most flexible)

What it means: this is classic git rebase -i with an edit action — git "rewinds" the branch to the bad commit, pauses so you can change it, then replays the rest on top. Use it when the fix is more involved than deleting lines (e.g. you also want to rewrite them).

  1. Commits panel → select the bad commit → e. The rebase pauses with that commit as the latest one.
  2. Fix the file in your editor (e on it in the files panel), then stage the fix and press A to amend it into the (now-latest) commit.
  3. m → continue the rebase. The remaining commits are replayed on top of the corrected one.

Method 3 — Fixup commit (rewrites history; good for review-then-squash flows)

What it means: instead of touching the old commit directly, you commit the correction now as a special fixup! commit "addressed to" the old commit, and squash it in afterwards (git's --autosquash). Nice because you can batch several corrections before rewriting once.

  1. Fix the lines in your editor, then stage just that change (files panel → enter the file → space on lines).
  2. Commits panel → select the original commit → F to create a fixup! commit. (Shortcut: ctrl+f in the files panel finds the right target commit for you automatically.)
  3. When ready, select the original commit and press S — all its fixup! commits above get squashed in, as if the mistake never happened.

Method 4 — Forward fix: reverse the lines in your working tree (no rewrite; safest)

What it means: history is left alone. You change the lines back to how they were and commit that as a new commit. The mistake and its undo are both visible in the log — which is exactly what you want on a shared or pushed branch. Two ways to get the old lines back without retyping them:

  1. Reverse-apply a patch: build the patch exactly as in Method 1 (commit → file → select lines → space), but at the ctrl+p menu choose apply patch in reverse. The lines flip back to their previous state in your working tree only — no commits touched.
  2. — or — Checkout the old file version: select the commit before the bad one → enter → select the file → c to check out the file as it was at that commit. This restores the whole file as it was, so next: files panel → enter the file → stage only the lines that undo the mistake → d to discard the rest.
  3. Commit as normal (c) — e.g. "undo debug logging from abc123". Push without force.

Method 5 — Revert the whole commit, then trim (no rewrite)

What it means: t on a commit is git revert — it creates a new "anti-commit" containing the exact inverse of every change in the original. Perfect as-is when the whole commit was bad. If only some lines were bad:

  1. Commits panel → select the bad commit → t to revert it (the anti-commit appears at the top).
  2. Select the new revert commit → g → mixed reset to the commit below it. The revert commit is un-committed and all its reversals become unstaged changes.
  3. Files panel → keep only the reversals you want (stage those lines), discard the rest with d, and commit.

Which one should I use?

MethodRewrites history?Best when
1. Custom patchYes — force-push if pushedPure line deletion on a branch only you work on; fewest steps, all inside lazygit
2. Edit mid-rebaseYesThe fix needs real editing, not just removing lines
3. Fixup commitYes (at squash time)You want to batch corrections, or squash later after review
4. Forward fixNoBranch is shared or already pushed — the default safe choice
5. Revert + trimNoMost of the commit was bad anyway, or you want an explicit undo in the log
⚠️ Rewriting history (methods 1–3) gives commits new SHAs, so if the branch was already pushed you must force-push (P — lazygit offers --force-with-lease, which aborts if someone else pushed in the meantime). Anyone else with the branch checked out will have to reset onto it. On shared branches prefer methods 4–5. And if anything goes sideways: z walks the reflog and undoes it.

Handy workflows

  • Amend an old commit: stage the changes (files panel) → commits panel → select the commit → A. Or ctrl+f in the files panel to auto-find the target and create a fixup.
  • Split the last commit: commits panel → g → soft reset to the commit before → restage and commit in pieces.
  • Undo a bad rebase: z (repeatedly) — it walks the reflog.
  • See what git commands lazygit runs: @ → toggle command log; great for learning the underlying git.
  • Diff two branches: W on the first ref → move to the second → W → view diff; esc exits diff mode.
  • Bisect: commits panel → b on a bad commit → mark good/bad as it steps.

Glossary — what these things actually mean

TermMeaning
Working treeThe files on disk as you currently see them — your editable copy of the project.
Index / staging areaThe draft of your next commit. space moves changes between the working tree and the index; only staged changes end up in the commit.
CommitA permanent snapshot of the whole project, identified by a SHA hash. Commits are never edited in place — "changing" one really means replacing it with a new commit.
HEADThe commit you currently have checked out — usually the tip of your branch.
PatchA list of line-level changes (a diff) that can be applied to code — or applied in reverse to undo those same lines. Lazygit's custom patch builder lets you assemble one interactively from a commit.
HunkOne contiguous block of changed lines within a diff. Staging "by hunk" (a) grabs a whole block at once instead of line by line.
Interactive rebaseGit rewinds your branch and replays its commits one at a time, letting you reorder, drop, squash, or edit each along the way. Nearly all of lazygit's history-editing keys are shortcuts that drive one of these behind the scenes.
Rewriting historyReplacing existing commits with new ones (new SHAs). Harmless on a branch only you use; on a pushed branch it requires a force-push and disrupts anyone else tracking that branch.
RevertA new commit containing the exact inverse of an earlier commit's changes (t). Undoes the effect without rewriting history.
ResetMoves your branch pointer to an earlier commit. Soft: later changes stay staged. Mixed: they stay in the working tree, unstaged. Hard: they're discarded entirely.
Fixup commitA commit whose message starts with fixup!, marking it as a correction destined to be squashed into a specific earlier commit (F to create, S to squash them in).
SquashCombining two or more commits into one. "Squash" keeps both messages; "fixup" keeps only the original's.
Cherry-pickCopying a commit's changes onto another branch as a new commit (C then V), without merging the whole branch.
StashA shelf for uncommitted changes: s sets them aside to get a clean working tree, and they can be re-applied later — even on another branch.
Force-pushOverwriting the remote branch with your rewritten local one. Lazygit uses --force-with-lease, which refuses if someone else pushed in the meantime — always prefer it over a plain force.
ReflogA local journal of every position HEAD has been at. It's why z can undo "destructive" operations like resets and botched rebases — the old commits are still reachable.
Merge conflictWhen two changes touch the same lines and git can't pick a winner, it stops and asks you to choose (the conflicts view). After resolving, m continues the operation.
UpstreamThe remote branch your local branch tracks — what p pulls from and P pushes to.
Detached HEADChecking out a commit directly instead of a branch (space on a commit). Fine for looking around; create a branch (n) before committing new work there.
Defaults verified against lazygit's pkg/config/user_config.go. Your own config lives at the path shown by e in the status panel.
