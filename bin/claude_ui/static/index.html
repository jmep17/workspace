~/.claude/config

Skills, commands, agents, and config files — versioned here, symlinked into the Claude config dir. Collections (like work/, gitignored) mirror the same shape and merge in automatically. Everything is plain files: review & commit with git — or use the git panel below. Ctrl/⌘-K opens the command palette.

how this works

The repo is the source of truth; the config dir just points at it. Claude Code reads one config dir (~/.claude by default). The links panel symlinks each piece of it — skills, commands, agents, output-styles, CLAUDE.md, settings.json, keybindings.json — to files in this repo, so your real config lives in git and a git pull updates every machine.

Importing an existing config: setting "config dir" to a path does NOT import anything by itself — it only chooses where the symlinks are managed. But when a row shows "real", clicking link takes over that content: if the repo side doesn't exist yet it is adopted (moved into the repo, then symlinked back — commit it after); if the repo side already exists, the old content is renamed to *.bak next to it and the link replaces it. Alternatively, ⇪ folder-upload any config-shaped directory (containing skills/, commands/, CLAUDE.md, ...) to import it as a collection without touching your live config.

Collections are extra config sets at the repo root (e.g. work/). Their skills/commands/agents/output-styles merge into the live config automatically via generated symlinks (a collection command runs as /<collection>:<name>; a collection skill loads as <collection>-<name>). Config files can't merge — Claude Code reads exactly one CLAUDE.md / settings.json — so each file row has a source dropdown choosing which copy is linked on this machine. work/ is gitignored: private, one folder, never committed.

MCP servers are different: Claude Code stores user-scope servers in ~/.claude.json (machine state, NOT inside the config dir, and not affected by the config-dir setting). The mcp tab keeps server definitions versioned in the repo (claude/mcp-servers.json or a collection's copy) and applies them into ~/.claude.json per machine; machine-only servers can be adopted into the repo or removed.

Statusline: every field documented in the statusline stdin schema is available from the palette on the right, grouped by category — click or drag a field onto a line, drag chips to reorder or move them between lines, and drag one back to the palette (or hit its ×) to remove it. Up to four lines are supported (narrow terminals truncate a long single line). Click a chip to pick its colour — one of the palette presets or a custom hex colour (custom needs a truecolor terminal) — or toggle bold. Preset layouts give a quick starting point. A visible separator automatically gets a space on each side and renders dimmed so the fields stand out. Claude Code re-runs the script after each assistant message — set a refresh interval to also update while idle; no restart is ever needed. The cost today / 7 days / month fields shell out to ccusage (or npx ccusage) in the background and cache results for 5 minutes — they stay hidden until the first refresh completes, and show nothing if ccusage isn't available.

Limitations: a non-default config dir only takes effect if you export CLAUDE_CONFIG_DIR=... in your shell — the UI can't do that for you. Claude Code reads settings/CLAUDE.md at session start, so changes apply to new sessions. Skills must be direct children of the skills dir (that's why nested folders get generated links). Moving a skill into a folder renames it (tdd → ai-tdd). Nothing here touches project-level config (.claude/, .mcp.json inside your projects). This page runs locally from your clone and edits plain files — nothing is committed behind your back; the git panel up top shows what changed, with per-file diffs and an explicit commit button.
