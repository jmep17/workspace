Glossary

Shared vocabulary for the course. Every lesson uses these terms in exactly these senses.

Port
A 16-bit number (0–65535) that lets one machine host many network conversations at once. "Port 3000" means nothing by itself — it's always half of an (address, port) pair.
Socket
The OS object a program uses to send and receive network traffic. A listening socket is one that has claimed an (address, port) pair and waits for incoming connections.
Bind
The system call that claims an (address, port) pair for a socket. This is the moment ownership is decided — before any traffic flows. See Beej's Guide.
EADDRINUSE
The error the OS returns when a program tries to bind an (address, port) pair that another listening socket already owns. This is what a second npm run dev on port 3000 dies with.
Loopback
The virtual network interface (127.0.0.1, and the whole 127.0.0.0/8 range) that never leaves your machine. localhost is a hostname that resolves to it.
Origin
A browser-security triple: scheme + host + port. http://localhost:3000 and http://localhost:3001 are different origins — cookies and storage are not shared between them.
Client (OAuth)
Your application, as registered with the authorization server. Defined in RFC 6749.
Authorization server / IdP
The service that authenticates the user and issues authorization codes/tokens (Auth0, Google, Okta, an internal identity provider…). It is the party enforcing your redirect_uri rule.
redirect_uri (redirection endpoint)
The URL the authorization server sends the user's browser back to after login, carrying the authorization code. Registered ahead of time; validated on every request. RFC 6749 §3.1.2.
Exact string matching
The validation rule from RFC 9700 §2.1: the presented redirect_uri MUST equal a registered one character-for-character (RFC 3986 §6.2.1 simple string comparison). No wildcards, no "close enough", no dev-mode leniency.
Loopback port exception
The single carve-out to exact matching: native apps (desktop/mobile binaries) may vary the port on loopback redirect URIs. RFC 8252 §7.3. Web apps do not qualify.
Reverse proxy
A server that owns a front-door (address, port) and forwards each incoming request to one of several backend servers, relaying the response. The browser only ever sees the front door.
Upstream
One of the backend servers a reverse proxy forwards to. In this course: one running branch = one upstream on its own internal port.
X-Forwarded-* headers
Headers a proxy adds to the copied request so the upstream can see past it: X-Forwarded-For (real client address) and X-Forwarded-Proto (original scheme). Added by Caddy's reverse_proxy by default.
Automatic HTTPS (Caddy)
Caddy serves TLS for any site address with a recognizable hostname — including localhost — unless the address carries an explicit http:// prefix. Relevant here because the scheme is part of exact matching. Docs.
