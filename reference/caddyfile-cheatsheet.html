Caddyfile cheat-sheet

Only what this course uses. Full docs: caddyserver.com/docs/caddyfile

The doorman pattern

# Caddyfile
http://localhost:3000     # site address — explicit http:// or Caddy auto-serves HTTPS

reverse_proxy :3101       # upstream: the branch that is currently "live"

Run · reload · inspect

caddy run --config Caddyfile      # foreground; Ctrl-C stops
caddy reload --config Caddyfile   # hot-swap config, zero downtime
caddy validate --config Caddyfile # syntax check without running

lsof -nP -iTCP:3000 -sTCP:LISTEN  # who owns the front door?

Equivalent one-liner (no Caddyfile)

caddy reverse-proxy --from :3000 --to :3101
# bare :3000 (no hostname) also stays plain HTTP

Headers the proxy adds to the copied request

Host: localhost:3000        # passed through unchanged
X-Forwarded-For: 127.0.0.1  # the real client's address
X-Forwarded-Proto: http     # the scheme the browser used

Docker variant

docker compose up                 # from labs/doorman-docker/
docker compose exec doorman caddy reload --config /etc/caddy/Caddyfile
docker compose down
  • Upstreams by service name (stable:3101) — in-container localhost is the container itself.
  • Host processes are reached as host.docker.internal:PORT.
  • Publish the front door on loopback only: "127.0.0.1:3000:3000".

Gotchas (each learned in a lesson)

  • Bare hostname ⇒ automatic HTTPS. localhost:3000 as a site address serves TLS; the scheme then fails exact-match validation. (Lesson 0002, docs)
  • One listener per (address, port). Stop any dev server squatting on 3000 before caddy run, or you'll get the lesson-1 error: EADDRINUSE. (Lesson 0001)
  • reload needs the admin endpoint. It talks to the running Caddy on localhost:2019 — so it only works while caddy run is up.
